# Keycube Heatmap 3D

Interactive 3D visualization of Keycube study data.

The project currently exposes two autonomous views:

- `Preference`
- `Reachability`

The root URL `/` redirects to `/preference/`.

Public demo: <https://keycube.github.io/heatmap3d>

## Project overview

`heatmap3d` is a static web application used to explore Keycube study data directly on a 3D representation of the device.

Current stack:

- HTML with Jekyll-style layouts and includes
- Vanilla JavaScript
- Three.js loaded from `unpkg.com`
- CSV datasets transformed into browser-usable data at build time
- Local static build generated by `scripts/build_local_site.py`

## Features

- `Preference`: participant-by-participant preference data, plus aggregate view
- `Reachability`: per-finger and combined reachability heatmaps
- 3D key visualization driven directly by study datasets
- English and French page variants

## Requirements

### Prerequisites for the recommended local build

To build and run the app locally with the recommended workflow, you need:

- Python 3.10+ recommended
- the Python standard library modules used by `scripts/build_local_site.py`
  - `csv`
  - `json`
  - `re`
  - `shutil`
  - `pathlib`
- permission to create and overwrite the generated folder `.local_site/`
- a free local port for the HTTP server
  - examples in this README use port `4000`
- a modern browser with:
  - ES modules support
  - import map support
  - WebGL enabled
- Internet access at runtime for:
  - `https://unpkg.com/three@0.160.0/build/three.module.js`
  - `https://unpkg.com/three@0.160.0/examples/jsm/`

Notes:

- no Node.js dependency is required for the recommended local build
- no Python package installation is required for the recommended local build
- the local build script writes generated output to `.local_site/`

### Optional Jekyll prerequisites

Jekyll is not the recommended local workflow here, but if you want to use it, you need:

- Ruby 3.x
- Bundler
- network access to `rubygems.org`
- a Ruby environment compatible with the gems declared in `Gemfile`

The recommended path for local work in this repository is the static Python build, not Jekyll.

## Run locally

From the repository root:

```bash
python3 scripts/build_local_site.py
python3 -m http.server 4000 --directory .local_site
```

What these commands do:

- `python3 scripts/build_local_site.py`
  - reads the source HTML, layouts, includes, CSS, JS, and CSV files
  - generates a complete static site into `.local_site/`
- `python3 -m http.server 4000 --directory .local_site`
  - serves the generated static site on `http://127.0.0.1:4000`

Open:

- <http://127.0.0.1:4000/>
- <http://127.0.0.1:4000/preference/>
- <http://127.0.0.1:4000/reachability/>

Behavior:

- `/` redirects to `/preference/`
- `/preference/` serves the autonomous Preference page
- `/reachability/` serves the autonomous Reachability page

## Maintenance workflow

The local server reads generated files from `.local_site`.

After modifying source files, regenerate the local output:

```bash
python3 scripts/build_local_site.py
```

Rebuild after changes to:

- `index.html`
- `preference.html`
- `reachability.html`
- `preference-fr.html`
- `reachability-fr.html`
- `_layouts/`
- `_includes/`
- `_data/*.csv`
- `assets/css/`
- `assets/js/`
- `scripts/build_local_site.py`

If the server is already running, you usually do not need to restart it. A hard refresh in the browser may still be necessary.

## Key files

### Pages

- `index.html`: technical redirect
- `preference.html`: preference visualization UI
- `reachability.html`: reachability visualization UI

### Layouts

- `_layouts/dataviz.html`: shell for interactive 3D pages
- `_layouts/default.html`: minimal layout for simple pages and redirect flow

### Includes

- `_includes/model-viewer.html`: main interactive Three.js scene
- `_includes/dataviz-data.html`: injects CSV-derived data into the page
- `_includes/model-viewer-static.html`: older static viewer include, not central to the current flow

### Frontend

- `assets/js/dataviz.js`: main UI logic
- `assets/css/dataviz.css`: styling for `Preference` and `Reachability`
- `assets/css/default.css`: minimal styles for simple pages

### Data

- `_data/preferences.csv`: participant preference dataset
- `_data/reachability.csv`: per-finger reachability dataset

### Local build

- `scripts/build_local_site.py`: builds a directly servable static site into `.local_site`

## Data flow

1. Source datasets live in `_data/`
2. Templates and includes define the page structure
3. `scripts/build_local_site.py` renders a static local site into `.local_site`
4. The browser loads generated pages
5. Three.js is loaded through the import map
6. `assets/js/dataviz.js` drives interactions using injected data

## Troubleshooting

### Local server runs but changes are not visible

- rerun `python3 scripts/build_local_site.py`
- hard refresh the browser
- make sure you are looking at `http://127.0.0.1:4000/`

### The 3D cube does not render

- check the browser console
- confirm WebGL is enabled
- confirm `unpkg.com` is reachable

### Data looks wrong

- verify CSV column names
- verify `_includes/dataviz-data.html`
- verify `assets/js/dataviz.js`

### `python3` is not available

- install Python 3
- rerun the commands with the correct Python binary for the machine

### Jekyll does not start

Jekyll is not the recommended local workflow here.

If you still want to use it, you will need:

- Ruby 3.x
- Bundler
- `bundle install`
- a Ruby environment compatible with the gems in the repository
